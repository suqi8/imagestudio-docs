# 工作台与插件

## 工作台

当用户从工作区进入某个项目后，会进入 `Workbench(projectName)`。

这是项目上下文下的实际操作区域，适合承载：

- 镜像解包
- 文件浏览与修改
- 打包与导出
- 日志查看
- 插件执行

## 插件入口

项目中已经有独立的 `PluginScreen(projectName)`，说明插件并不是全局无上下文执行，而是与当前项目工作流相关联。

这类设计有几个优点：

- 插件可以直接感知当前项目
- 权限与能力范围更容易控制
- 插件 UI 可以围绕工作台操作组织

## 当前插件方向

从现有 RFC 与代码结构看，插件系统的目标不是脚本拼接，而是 **原生插件宿主体系**，重点包括：

- `.isp` 作为插件分发包
- `plugin.toml` 作为元数据入口
- Compose UI 集成
- 权限模型
- 后台任务能力
- 多平台宿主兼容

## 对公开文档的建议

如果你的文档站对外公开，这一页建议保持“可理解但不过度泄露实现细节”的风格：

- 对用户讲清楚插件是什么、能做什么
- 对开发者引导到插件系统章节
- 对未完成能力明确标注“规划中”或“实验性”
